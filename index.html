Architecture & Data Model

This pillar defines the logical architecture, relational data model, multi-tenancy and RBAC enforcement, API surface, Redis and object-storage flows, and a capacity-sized deployment topology for the Praveno Platform. All designs are tenant-scoped by organization_id, defense-in-depth aligned to NIST SP 800-53 Rev 5 control families (notably AC, AU, SC, SI, CM), and traceable to the NIST SP 800-37 Rev 2 RMF steps (Prepare, Categorize, Select, Implement, Assess, Authorize, Monitor). Per FIPS 199 the system is provisionally categorized Moderate (Confidentiality=Moderate, Integrity=Moderate, Availability=Moderate) given business program data and PII fields; the high-water mark drives control selection.

1. Logical Architecture Overview

Five tiers, each independently scalable and observable. The design is stateless at the application tier so any request can hit any node (session/state externalized to Redis and PostgreSQL), which is what makes horizontal autoscaling and blue/green deploys safe.

3. Multi-Tenancy & RBAC

Isolation is defense-in-depth: (a) application layer sets tenant context on every connection, and (b) PostgreSQL Row-Level Security (RLS) enforces it in the database so a query bug cannot leak cross-tenant rows. This directly implements AC-3 (access enforcement), AC-4 (information flow enforcement), and SC-4 (information in shared resources).

-- The API sets these per request/transaction after authenticating & authorizing:
--   SET LOCAL app.current_org  = '';
--   SET LOCAL app.current_role = '';

ALTER TABLE tasks ENABLE ROW LEVEL SECURITY;
ALTER TABLE tasks FORCE ROW LEVEL SECURITY;   -- applies even to table owner

-- Tenant isolation: rows are only visible within the caller's org.
CREATE POLICY tenant_isolation ON tasks
  USING (organization_id = current_setting('app.current_org')::uuid);

-- Write guard: viewers cannot mutate; inserts must carry the caller's org.
CREATE POLICY tenant_write ON tasks
  FOR ALL
  USING  (organization_id = current_setting('app.current_org')::uuid)
  WITH CHECK (
     organization_id = current_setting('app.current_org')::uuid
     AND current_setting('app.current_role') <> 'viewer'
  );

Field-level restriction on sensitive columns. Columns such as users.compensation, users.national_id, or memberships.pay_band are protected beyond row-level rules: (1) held in a separate user_sensitive table with its own RLS restricting SELECT to Owner/Admin; (2) exposed only through a SECURITY DEFINER view / GraphQL field-level authorization that returns null for unauthorized roles; (3) encrypted at rest with a column key (SC-28, SC-12 key management) and never written to audit_logs.diff in cleartext (redaction filter). This satisfies AC-6 least privilege at field granularity.

4. API Design

REST is the primary transactional surface; GraphQL serves rich read/aggregation for dashboards to avoid client-side over-fetching; webhooks push events to integrations. All endpoints require OIDC/JWT bearer auth, are scoped by OAuth2 scopes, and inherit RLS tenant context. Input validated against schema (SI-10), and all mutating routes are rate-limited (SC-5) and audit-logged (AU-2). Endpoints map to OWASP ASVS: V1 (Architecture), V4 (Access Control), V5 (Validation/Encoding), V7 (Error/Logging), V13 (API/Web Service) in ASVS v4.0.3; equivalently the V1–V17 chapters in ASVS v5.0. Mobile clients additionally meet OWASP MASVS (MASVS-AUTH, MASVS-NETWORK, MASVS-STORAGE).

Security & Controls

This pillar defines the security control baseline for the Praveno Platform (multi-tenant SaaS) and its HTML5 PWA / Android / iOS proof-of-concept. Controls are mapped to FIPS 199 categorization, NIST SP 800-37 Rev 2 Risk Management Framework, NIST SP 800-53 Rev 5, OWASP ASVS (target Level 2), and OWASP MASVS v2. The production stack is fixed (PostgreSQL, Redis, S3-compatible object storage, REST + GraphQL + webhooks); every control below is designed around that stack.

Scope & assumptions. Multi-tenant, cloud-hosted, internet-facing web + mobile application processing organizational business-workflow data (project/task/approval records, comments, file attachments, PII of workforce users, and — for regulated customers — potentially Controlled Unclassified Information (CUI)). Sizing target: 20,000 users across Web (PWA), Android, and iOS. Data is not classified national-security information; the highest sensitivity handled is CUI + business-confidential + PII.

1. FIPS 199 Security Categorization

Per FIPS PUB 199, the potential impact of a loss of Confidentiality, Integrity, or Availability is rated LOW / MODERATE / HIGH. The overall category is the high-water mark across the three objectives.

Security ObjectiveImpact LevelRationale
ConfidentialityModerateSystem stores workforce PII (names, emails, roles), business-confidential project data, approval/routing decisions, and sensitive records entrusted by tenant organizations. Unauthorized disclosure of one tenant's data to another would cause serious adverse effect on organizational operations, reputation, and contractual/legal standing — but not catastrophic/national-security harm.
IntegrityModerateApprovals, RACI assignments, task-state transitions, exports, and audit logs are decision-of-record. Undetected modification (e.g., a forged approval or altered due date) could cause serious adverse effect on program delivery and accountability. Integrity is enforced by RLS, immutable append-only audit logs, and record hashing.
AvailabilityModerateField users depend on the platform for daily task and approval workflows; extended outage causes serious operational disruption and SLA breach. However, business workflows tolerate short (minutes-to-hours) disruption and offline PWA mode provides degraded continuity, so impact is not High.
Overall categorization: SC Praveno = {(Confidentiality, MODERATE), (Integrity, MODERATE), (Availability, MODERATE)} → System Impact Level = MODERATE. This selects the NIST SP 800-53 Rev 5 MODERATE baseline (via SP 800-53B) as the tailoring starting point. FedRAMP Moderate is the reference profile if a government or regulated tenant requires an authorization.

2. Capacity Model (explicit math & assumptions)

Stated assumptions. (A) 20,000 registered users. (B) Peak concurrency = 12% — a defensible figure for a B2B workflow tool where usage clusters in business hours across a few time zones; we also stress-test 15% as an upper bound. Narrative. At the 12% peak, the platform runs at ~240 req/s. A four-node app tier (two mathematically required, doubled for N+1 and multi-AZ availability, satisfying CP-2/SC-6) sits at only ~24% CPU, leaving room to absorb the ~400 req/s spike within a single tier and the 15%/3,000-session stress case with autoscaling. The database is the true bottleneck: with transaction-mode PgBouncer, 800 hot application connections collapse onto ~100 Postgres backends, keeping max_connections at a safe 200 with headroom for replicas and admin. Redis is effectively free at this scale — sized instead for durability (AOF everysec) and HA (Sentinel/Cluster) rather than throughput. Object-storage bytes never traverse the app tier because uploads/downloads use pre-signed URLs (SC-8 in transit, virus-scan-on-write), so the ~576 Mbps of file traffic is absorbed by S3, not the compute layer.

(c) Distributed load generation — how to actually reach 20,000 VUs
A single k6 or Artillery process cannot sustain 20,000 VUs — one machine tops out around 1,000–3,000 VUs before the generator (CPU, ephemeral ports, file descriptors) becomes the bottleneck and corrupts results. To reach true 20k you MUST distribute: run k6 Cloud (managed, geo-distributed) or the k6 Operator on Kubernetes (parallelism: 20 shards × ~1,000 VUs each), or a fleet of Artillery/Locust workers behind a coordinator. Distribute generators across multiple regions to mirror the Web/Android/iOS geographic mix, raise ulimit -n to 250k+ and widen net.ipv4.ip_local_port_range on each agent, and aggregate metrics via Prometheus remote-write so per-shard results roll up into one SLO view. Always test against a production-parity staging tier (same node count, same PgBouncer/Postgres sizing) or the numbers are meaningless.

Mobile & DevSecOps

This pillar defines how the single web build of the Praveno Platform is delivered to three targets — Web (PWA), Android, and iOS — from one codebase, how field users stay productive on degraded or absent connectivity, and how every artifact is proven secure before release. Design goal: write once, package thrice, sign everything, and preserve audit integrity even when the device is offline. Security controls trace to NIST SP 800-53 Rev 5 (families SI, SC, SA, CM, AU, RA), OWASP ASVS v4.0.3/v5.0, and OWASP MASVS for the packaged mobile apps.

Single-source packaging model. The PoC produces one dist/ web build. Android wraps it as a Trusted Web Activity (TWA) or a Capacitor WebView app; iOS wraps it as a Capacitor / WKWebView app. No forked UI code — only a thin native shell plus platform plugins for push, biometrics, and secure storage.

1.2 Service worker caching strategy

Three layered strategies: (a) app-shell precache — the HTML/JS/CSS/icon core is cached on install so the app boots with zero network; (b) stale-while-revalidate for read data — GET responses for reference/list data are served instantly from cache while a background fetch refreshes the entry; (c) background sync for mutations — writes made offline (create task, approve, comment) are enqueued and replayed by the sync event once connectivity returns. Navigation requests fall back to a cached offline.html app shell. HTML is network-first to avoid serving a stale shell; static assets are cache-first.

const SHELL = 'praveno-shell-v7';
const DATA  = 'praveno-data-v7';
const SHELL_ASSETS = ['/', '/index.html', '/offline.html',
  '/app.css', '/app.js', '/icons/icon-192.png', '/manifest.webmanifest'];

self.addEventListener('install', (e) => {
  e.waitUntil(caches.open(SHELL).then((c) => c.addAll(SHELL_ASSETS))
    .then(() => self.skipWaiting()));
});

self.addEventListener('activate', (e) => {
  e.waitUntil(caches.keys().then((keys) => Promise.all(
    keys.filter((k) => ![SHELL, DATA].includes(k)).map((k) => caches.delete(k))
  )).then(() => self.clients.claim()));
});

self.addEventListener('fetch', (event) => {
  const { request } = event;
  if (request.method !== 'GET') return; // mutations go through Background Sync queue

  const url = new URL(request.url);

  if (request.mode === 'navigate') {            // network-first HTML, offline fallback
    event.respondWith(fetch(request).catch(() => caches.match('/offline.html')));
    return;
  }
  if (url.pathname.startsWith('/api/')) {       // stale-while-revalidate for data
    event.respondWith(caches.open(DATA).then(async (cache) => {
      const cached = await cache.match(request);
      const network = fetch(request).then((res) => { cache.put(request, res.clone()); return res; })
        .catch(() => cached);
      return cached || network;
    }));
    return;
  }
  event.respondWith(caches.match(request).then((c) => c || fetch(request))); // cache-first assets
});

self.addEventListener('sync', (event) => {      // Background Sync: replay queued mutations
  if (event.tag === 'praveno-outbox') event.waitUntil(replayOutbox());
});
1.3 Offline data (IndexedDB) & conflict resolution

Structured offline state lives in IndexedDB (via a thin wrapper such as idb): object stores for tasks, projects, comments, an append-only outbox of pending mutations, and a meta store holding the last syncCursor. Each record carries { id, rev, updatedAt, deleted }. Conflict handling is chosen per entity class:

StrategyWhere usedBehavior & trade-off
Last-write-wins (LWW)Low-contention scalar fields (status, due date)Highest updatedAt (server-authoritative timestamp) wins. Simple; risks silently discarding a concurrent edit.
Per-field mergeTask/project records with independent fieldsMerge non-overlapping field changes; only truly-conflicting fields need resolution. Preserves more user intent than record-level LWW.
Version vectorApprovals, checklists, RACI (multi-actor)Detects genuine concurrency (neither vector dominates) and surfaces a conflict for user/rule resolution instead of guessing. Higher metadata cost.
TombstonesAll deletesDeletes stored as deleted:true records, not physical removal, so a delete propagates and a stale offline client cannot "resurrect" a removed row. Tombstones are garbage-collected after the retention window.
Never trust the client clock for ordering. LWW uses the server-assigned updatedAt. The client's local timestamp is advisory only; the server re-stamps on ingest (see §4) to keep audit ordering and AU-8 time-stamp integrity correct.

3. iOS packaging

iOS has no TWA equivalent: Safari can install a PWA to the Home Screen, but a pure PWA cannot receive server push without a native APNs registration path, has tighter storage-eviction rules, and gives up native store distribution. Therefore iOS ships via Capacitor over WKWebView, wrapping the identical web build.

npm i @capacitor/ios
npx cap add ios
npm run build && npx cap copy ios
npx cap sync ios
npx cap open ios          # opens Xcode: set Team, Bundle ID, signing, capabilities
  • Signing & review. In Xcode set the Apple Developer Team, bundle id com.praveno.app, and enable Push Notifications + Background Modes capabilities. Archive, upload to App Store Connect, and submit. Reviewers apply Guideline 4.2 (minimum functionality) — a shell that is only a wrapped website risks rejection, so native push, biometrics, and offline behavior are what justify the app.
  • Push via APNs. Web-push is unavailable through the wrapper; use @capacitor/push-notifications registering device tokens with APNs, fanned out server-side through Redis notification queues.
  • Keychain for secrets. Tokens and refresh secrets live in the iOS Keychain (hardware-backed, Secure Enclave where available) via a secure-storage plugin — never in localStorage or WebView cookies. Aligns with MASVS-STORAGE.

4. Low-connectivity & field-user support

Field users at customer sites, warehouses and remote job sites often operate on intermittent or metered links. The client is built offline-first, and the server remains the sole authority for identity, ordering, and audit truth.

MechanismDesign
Request queueingMutations append to the IndexedDB outbox with a client-generated idempotency key (UUID). A Background Sync replayOutbox() drains them in order; the server dedupes on the key so a retry never double-applies.
Exponential backoffFailed replays retry with backoff + jitter (e.g. 1s, 2s, 4s, 8s… capped ~5 min) to avoid thundering-herd on reconnect. Persistent 4xx (validation) fail fast to a conflict queue; 5xx/network errors keep retrying.
Delta syncReads pull only records changed since the stored syncCursor (server change-token / updated_since), not full tables — minimizing bytes on metered connections. Cursor advances only after a fully persisted batch.
Cache-size limitsData cache and IndexedDB are bounded (LRU eviction, per-store quotas, attachment cache capped e.g. 50 MB). The app checks navigator.storage.estimate() and requests persistent storage to resist eviction; large attachments stream via signed URLs rather than being fully cached.
Offline audit integrityClient records an audit event locally (actor, action, client-time, idempotency key) but it is provisional. On sync the server re-validates authorization, assigns the authoritative event_id and server timestamp, and appends to the immutable audit_logs. The client clock is never the system of record.
Audit chain of custody. Because the server re-validates and re-timestamps every offline event (NIST AU-8 time stamps, AU-9 protection of audit information, AU-10 non-repudiation), an offline device cannot forge order, backdate, or bypass authorization. Idempotency keys give exactly-once semantics under retry.
